Saving Data to Firebase

There are three firebase functions that will add or modify your data:

var ref = firebase.database().ref('/some/path');
var obj = {someAttribute: true};

ref.push(obj);   // Creates a new ref with a new "push key"
ref.set(obj);    // Overwrites the path
ref.update(obj); // Updates only the specified attributes

ref.push

There are two ways to use ref.push(). You can use it to create a new ref with an automatically generated push key, or you can save data to a new push key in a single step.

Both of these methods create new refs with their own push keys. Push keys are a critical concept in Firebase, because they enable us to save collections of data and sort them by creation date without key collisions. This would be impossible using zero-indexed arrays like you commonly find in Javascript.

Let‚Äôs unpack that last paragraph. First, imagine a realtime game with millions of users pushing data to your app at the same time. If there are 500 records in a collection and two users try to simultaneously push data to that collection within the same millisecond, could Firebase handle that with a zero-indexed array? No! That would be a mess. You‚Äôd have two users trying to push to /somePath/501 at the same time ‚Äî a colliding keys disaster.

Instead of letting keys collide, Firebase enables us to create unique keys with ref.push(). These keys are fantastic in that they don‚Äôt collide while also being sortable by time. In case you missed that‚Ä¶ huge concept here‚Ä¶ the push keys are sortable by time. They‚Äôre generated using a timestamp plus some randomness to avoid collisions. Querying Data from Firebase

The Firebase SDKs for Node.js and the browser are very, very similar. We‚Äôll be able to copy/paste Node.js code directly into an HTML script tag and it will function the same. The difference between the two SDKs is how they handle authentication. Node.js auth is simple with a Google Cloud service account API key. You can‚Äôt put a Google Cloud service account in the browser without giving full read/write access to your Firebase to the world, so you‚Äôll need to authenticate using Google‚Äôs generic browser API before accessing your Firebase from the client.

Once authenticated, both Javascript SDKs expose the same API for interacting with Firebase, so we‚Äôll cover the API first in Node.js for simplicity.

Firebase events

Firebase delivers data to us with five different events accessed with ref.on(‚Äòevent‚Äô, callback):

  1. value

  2. child_added

  3. child_changed

  4. child_removed

  5. child_moved

These events are mostly self-explanatory, and they‚Äôre covered in the docs, but let‚Äôs get into the details.

ref.on(‚Äòvalue‚Äô, callback);

The value event fires once for an initial data load and again whenever any data changes. value is the event that we‚Äôll use most, typically with ref.once(‚Äòvalue‚Äô, callback), because it returns a ref‚Äôs children in a single operation. You can also use ref.on(‚Äòvalue‚Äô, callback) to keep an object in memory synced up with Firebase.

These events are less commonly used. Review child_changed, child_removed and child_moved in the docs. They do about what you‚Äôd think they do, but they‚Äôre mostly useful for advanced querying, so we‚Äôll cover them later as needed.

Use ref.off() to stop listening to a ref‚Äôs events. Just make sure to pass in the exact same callback function‚Ä¶ not a copy of the function.

The most common use of Firebase ref events is to listen to the value event for initial data load and then call .off on the listener as soon as the data has loaded. This is annoying to do over and over, so Firebase gives us ref.once(‚Äòvalue‚Äô, callback) to listen to the initial load and then turn off the listener automatically. ref.once() takes an optional callback and returns a promise. You can use *ref.once() with child_added, child_changed and child_removed *events, but it‚Äôs a less-common pattern.

You‚Äôll notice that ref.on requires a callback. DO NOT USE ref.on(‚Äòchild_added‚Äô).then(). It doesn‚Äôt make any sense to use ref.on() with a promise, because it is meant to fire over and over again like a normal event listener that you might use in the browser. ref.once() is a special case that returns a promise to make our lives easier.

Data snapshots

Firebase ref events return data snapshots. So far we‚Äôve just used snap.val() to get the snapshot‚Äôs JSON, but there are more snapshot functions‚Ä¶

  • snap.exportVal() ‚Äî Also gets the value (kind of duplicative)
